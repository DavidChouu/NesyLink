# 策略形式化与证明（技术附录 B）

## 1. 范围与代码对应

五个正式 Python Policy 位于 `examples/task1_agent.py` 至
`examples/task5_agent.py`。主要环境与五关策略形式化集中在
[`examples/NesyLinkEnvironment.lean`](examples/NesyLinkEnvironment.lean)；通用图搜索
理论由辅助文件 [`examples/BFSFormalization.lean`](examples/BFSFormalization.lean)
独立补充。

运行时 Policy 仅使用：

- 128×160 RGB `obs`；
- `safe_info["last_reward"]`；
- `safe_info["inventory"]`；
- 专用 `--task-policy` 可见但当前控制器不依赖的 `task_id`。

它们不读取 `room_id`、玩家坐标、真实 HP、对象列表、地图 JSON、动态机关状态、
事件或其他 full `info`。房间编号、对象位置和连接均由 CNN 符号结果及本局历史
动作建立。

Lean 验证的层次为：

```text
VerifiedObservation / ObservationRefinesWorld
             ↓
控制器记忆与高层目标选择
             ↓
有限 BFS / 短动作队列
             ↓
ControllerCommandSafe / 各 Task shield
             ↓
ClosedLoopStep / ClosedLoopExec
             ↓
EngineTick / EngineExec / Task Goal
```

CNN 本身、Python 容器实现和每一个像素中间帧不在 Lean 可信核内。

## 2. 通用证明架构

### 2.1 搜索与执行

通用搜索同时提供：

- 图路径和按深度可达性；
- 可计算 `executableBfsWithin`；
- 返回路径的起点、邻接、终点和安全性；
- 给定深度见证与 frontier 覆盖条件下的完备性；
- `TilePath`/`DirectionPlanSafe` 到 `Exec`、`EngineExec` 的动作翻译。

因此公开路线在源码中不是注释坐标序列，而是用关系项表达每一步方向、边界、目标
tile、交互前提和终态。是否已被 Lean 接受仍以整文件编译结果为准；本次复验的
两种规定命令均已退出 0，语法错误、编译错误、未解目标和待决元变量均为 0；
两份输出各有 79 条非致命 linter 风格建议，详见复现附件。动态任务仍需在每帧重新观测和过滤旧队列。

### 2.2 命令安全

`ControllerCommand` 区分普通导航、只转向、交互、攻击、举盾、宝箱 rush、出口
推进、memory fallback 和等待。`ControllerCommandSafe` 的义务随命令变化：

- 普通导航必须进入 `plannerSafe`；
- 只转向必须绑定前方真实宝箱或怪物；
- A 交互必须绑定可交互对象和正确优先级；
- 攻击必须有剑、正确朝向和目标怪物；
- rush 必须已举盾且物理可进入；
- 出口推进必须绑定真实出口、位置、方向和已满足条件；
- memory fallback 只能去安全格或已确认物理阻挡格，后者由碰撞安全停留。

### 2.3 闭环与完备性

`ClosedLoopStep` 将正确符号观测、安全命令、一次 `EngineTick` 和控制器更新装入
同一关系；`closedLoopExec_projects_engineExec` 证明闭环执行可投影到环境执行。

统一 `policy_complete_under_fairness` 使用自然数服务度量。它要求每个非零度量
状态都存在非空 `ClosedLoopExec` 严格下降，零度量推出目标。该条件概括最终正确
观测、有限前沿公平选择、怪物可再次接敌、交互成功和玩家存活等外部条件。它
不是无条件终止证明，也不把最终 `Goal` 或最终 `EngineExec` 作为前提。

## 3. Task1：静态记忆、FSM 与 BFS

### 3.1 Python 策略

阶段为：

```text
to_chest → to_exit → done
```

CNN 提供玩家、宝箱、墙、出口和危险 tile；控制器把历史宝箱/墙写入
`remembered_blockers`，避免开箱贴图变化后 BFS 把宝箱格当成地板。取得钥匙只
由允许的 inventory 确认。策略 BFS 到宝箱相邻格，短移动面向后按 A；随后 BFS
到锁门并沿门方向推进。所有队列动作再次经过 shield。

### 3.2 Lean 结论

- `remembered_blocker_is_retained`、`newly_remembered_blocker_is_present`：静态
  阻挡记忆单调且写入有效。
- `task1_phase_never_regresses`：阶段 rank 不下降。
- `task1Shield_allowed_move_is_enterable`：被放行移动物理可进入。
- `task1_two_phase_bfs_complete`：在有界可达与 frontier 条件下两阶段搜索完成。
- `task1_public_engine_certificate`：公开地图给出取得钥匙、首次消费钥匙并完成
  出口的闭合 `EngineExec`。

统一 `Task1.strategy_output_safe` 比其他关更强：由动作通过 shield 直接推出目标
tile 可进入。

## 4. Task2：可中断队列与有限战斗

### 4.1 Python 策略

阶段为：

```text
to_monster → to_chest → to_exit → done
```

怪物消失须连续三帧确认。移动队列不是盲目执行：每帧重新检查怪物、目标和下一
格，旧计划失效就清空。战斗在安全相邻攻击位进行，短移动只用于更新朝向；有剑
后按 A，每次命中伤害 1。非战斗阶段避开怪物及其邻域。

### 4.2 Lean 结论

- `three_missing_frames_confirm_clear`、`visible_monster_resets_missing_frames`；
- `task2_phase_never_regresses` 与队列延迟阶段切换；
- `attack_position_is_safe`、`face_monster_keeps_player_position`；
- `one_sword_hit_strictly_decreases`、`hp_sword_hits_are_sufficient`；
- `unsafe_queued_move_must_interrupt`、`interrupted_move_is_masked_to_wait`；
- Python parent 字典/小步 BFS 的 soundness 与有界完备性接口；
- 公开 HP=2 怪物的两次攻击、开箱、拿钥匙和条件出口完整证书。

动态完备性依赖怪物最终重新进入可攻击状态以及玩家能在有限攻击/等待中存活，
没有声称任意敌人随机轨迹都必然通关。

## 5. Task3：匿名房间图与反馈确认

### 5.1 运行时记忆

`Task3Agent` 不预设三房间名称或锁门方向。每个 `RoomMemory` 保存：

- 入口方向和双向 `connections`；
- 各方向出口种类与访问次数；
- 已观察宝箱、已确认开箱；
- 怪物出现/服务状态。

控制器还保存匿名 `current_room`/`next_room_id`、钥匙和剑反馈、待确认出口、待
确认交互、有限战斗失败次数、短移动及感知漏帧计数。

只有已有 `pending_exit_direction` 且前后玩家 tile 曼哈顿距离大于 3，才确认换房
并建立双向边。宝箱只由消失、钥匙增量或大额正奖励确认；怪物由消失和奖励
反馈更新。感知丢失时只允许有限复用出口推进或两帧局部移动，之后输出 WAIT。

### 5.2 目标优先级

```text
可见未开宝箱
→ 有剑处理怪物；无剑沿入口返回
→ 有钥匙服务当前或已知房间的锁门
→ 满足前置条件的未知出口
→ 房间图 BFS 回访可服务房
→ 入口回退/最少访问已知边/WAIT
```

房间图 BFS 返回到任一目标房间的第一条边，不能产生未学习连接。

### 5.3 Lean 结论

- `room_change_requires_pending_exit`、`no_pending_exit_cannot_confirm_change`；
- `confirmed_change_uses_existing_or_fresh_room`；
- `chest_confirmation_uses_only_public_feedback`；
- `locked_exit_requires_confirmed_key`；
- `perception_miss_beyond_exit_window_waits`；
- `graph_path_first_hop_is_learned`；
- `graph_path_is_invariant_under_room_renaming`；
- `mirror_preserves_selected_goal_category`；
- 公开地图从起始房到大厅战斗、钥匙房、返回并通过锁门的精确
  `public_map_complete_certificate`。

重命名和镜像定理约束目标类别与图结构，不声称 CNN 像素在任意镜像渲染下自动
正确。

## 6. Task4：未知桥指纹与观测驱动探索

### 6.1 通用策略

`RoomKind := unknown | switch | hub | leaf`。控制器依据当前可通行边界方向集合
形成 `BridgeFingerprint`，不假定桥有三态、固定顺序或固定目标房间。

换房时：有待定方向才建立双向边；没有方向的大跳变仅创建匿名节点，不伪造
连接。宝箱只在真正准备输出 A 的交互帧记为已服务。

### 6.2 目标顺序

```text
当前可见宝箱
→ 有剑攻击；无剑沿入口返回
→ switch 房先探索未知出口
→ 已连接后按需按 switch，并从当前真实可达出口离开
→ hub 选择当前指纹下可达且仍未服务的分支
→ 无候选时返回 switch 房
```

无钥匙时锁门不进入候选。出口按当前 BFS 距离选择，不使用固定方向优先级。

### 6.3 Lean 结论

- switch/hub/leaf 的视觉特征分类；
- `bridge_fingerprint_contains_exactly_reachable`；
- 指纹记忆与既有模式单调；
- 无方向跳变不产生虚假边，有方向新节点建立双向边；
- 可见宝箱最高优先、有剑才攻击、未知出口先于 switch；
- `generic_bridge_policy_does_not_assume_three_modes` 与四指纹回归；
- 通用公平性只要求有限指纹和重复按 switch 最终暴露可服务分支；
- 公开三态桥单独证明任意目标至多两次旋转，并构造取钥匙、取剑、击杀守卫、
  揭示/开启最终宝箱的完整 `EngineExec`。

三态旋转结论没有进入通用 Agent 定义。

## 7. Task5：反馈驱动探索与生存预算

### 7.1 相对记忆

Task5 保存相对房间坐标、父边、出口连接、宝箱/支持型宝箱、按钮、静态阻挡、
碰撞边、边代价、房间风险、怪物运动、威胁证据、已防护威胁格和每房服务战斗
尝试。房间关系来自实际出口推进和换房证据，不读取真实 room ID。

CNN 候选会结合玩家置信度、结构完整性、出口边界和运动连续性评分；墙体签名仅
用于纠正已观察房间。短动作队列每帧检查边界、阻挡边、动态威胁和目标状态。

### 7.2 公开反馈与生存预算

```text
documentedInitialHealth = 5
survivalBudget = 5 - observedDamage
rushMode ↔ survivalBudget ≤ 2
```

只有 `last_reward ≤ -1.0` 的已观察大额负反馈增加损伤计数，且上限为 5；步数
完全不参与 HP 推断。开箱由奖励或宝箱消失确认；若 key/gold 均无增量，则记录为
支持型宝箱并把损伤计数归零。

未知负奖励不会因为画面远处有怪物就归因给怪物。只有上一动作进入接触区、已经
接触，或连续视觉确认怪物向玩家移动时，才增强对应威胁证据。

### 7.3 实际目标顺序

Python `_choose_goal` 的顺序为：

1. 生存预算低、感知不确定、威胁证据充分且本房未尝试过时，进行一次有界阻挡
   怪战斗；
2. 有安全路径的未开宝箱；
3. 有盾时的直接宝箱路径；
4. rush 模式保留必需宝箱目标；
5. 为宝箱服务的有目的阻挡怪战斗；
6. 可达未按按钮；
7. 已完成叶子房沿父边回退；
8. 满足钥匙/按钮条件、按路径成本和风险排序的前沿出口；
9. 普通父边回退；
10. 房间图 BFS 回访仍有未开宝箱的房间；
11. WAIT。

### 7.4 六类安全模式

Lean `MoveMode` 为 normal、combat、chestRush、exitPush、shieldPrelude、
memoryFallback。普通导航必须进入 `safeTile`；战斗/rush 必须满足物理可进入及
怪物或盾牌条件；出口推进绑定满足条件的真实出口；fallback 只允许 WAIT、安全
格或可证明的阻挡停留。

Agent 视觉上可能把已开宝箱当地板，但引擎仍阻挡。Lean 证明首次尝试只产生位置
不变的 `moveBlocked`；碰撞反馈写入 `failedMoves/blocked_edges` 后，
`learned_blocked_edge_is_not_selected_again` 排除同一边再次进入搜索。

### 7.5 Lean 结论

- 相对连接证据可靠，visited/opened/pressed/failed 集合单调；
- 损伤上界、预算单调、支持宝箱重置、预算与 step count 无关；
- 远处或无关损伤不增加威胁，接触上下文允许归因；
- `bounded_blocker_combat_has_highest_priority`、
  `safe_chest_precedes_button`、`rush_preserves_required_chest_goal`；
- 六类输出谓词及 memory fallback 的环境可执行性；
- 房间图可达和公平服务的条件完备性；
- 公开四房间证书逐段执行中心宝箱、按钮、南房钥匙、消费钥匙进入东房、治疗
  宝箱、西房金币宝箱和最终全宝箱完成。

## 8. 五关统一验收接口的准确含义

每个 `TaskN` namespace 导出以下名字：

| 定理 | 准确含义 |
|---|---|
| `environment_execution_safe` | 在 `TickInvariantClosed` 的有限闭合系统中，`EngineExec` 保持 `WorldInvariant` |
| `strategy_output_safe` | Task1/2 从 shield 前提推出目标安全；Task3/4/5 从 `PolicyRefinesAgent` 推出实际决策输出的 `CommandSafe`/`PolicyOutputSafe` |
| `policy_refines_agent` | 反演 `ClosedLoopStep`，得到观测精化、安全命令、声明动作和 `EngineTick` |
| `policy_complete_under_fairness` | 局部非空闭环进展严格降低自然数度量时，存在完成闭环轨迹并投影为 `EngineExec` |
| `public_map_complete_certificate` | 存在公开初态到 `WorldCompleted ∧ alive ∧ ValidState` 的具体 `EngineExec` |

Task3/4/5 的统一 `strategy_output_safe` 不再接受并原样返回安全谓词；它们从各关
`PolicyRefinesAgent` 的目标选择与安全过滤合取证明中，提取实际 `PolicyDecision`
输出的安全性。其实质义务由各关的 `navigation_command_has_engine_step`、目标优先级、
BFS、交互和阻挡反馈等具体引理支撑。这仍不意味着“任意 Python 输出自动安全”。

同理，`policy_refines_agent` 是 Lean 闭环关系的投影定理，不是 Python 源码与
Lean `PolicyKernel` 的逐行语义等价证明。

## 9. 公开实例与运行时泛化

公开地图坐标只用于离线实例：

```text
具体 JSON 世界满足通用环境语义
→ 每段方向路径可安全执行
→ 交互/战斗/机关/出口前置条件成立
→ EngineExec 到达完成终态
```

Python 运行时目标选择不导入地图 JSON，也不使用公开房间 ID。Task3 匿名图重命名
与镜像、Task4 通用桥指纹、Task5 相对房间图用于说明策略结构不依赖固定命名；
它们不能替代正式 spatial/color 黑盒评测，实际泛化性能以
[`report_artifacts/robustness_suite_eval.json`](report_artifacts/robustness_suite_eval.json)
为准。

## 10. 证明强度与限制

源文件给出的证明项覆盖：环境基本不变量、BFS 健全性和有界条件完备性、命令/
动作过滤、阶段与记忆性质、关键交互、战斗 HP 下降、匿名图/桥指纹/生存预算性质、
闭环投影、局部公平条件下的终止，以及五张公开地图的具体完成轨迹。

没有声称：CNN 永远正确、随机怪物任意行为下无条件通关、浮点像素轨迹逐帧等同
tile 轨迹、Python 每行代码与 Lean 定义双向等价，或统一包装定理比其签名更强。

按既定行首统计口径生成的 472 个主要 `theorem` 声明见
[`report_artifacts/theorem_inventory.md`](report_artifacts/theorem_inventory.md)，
编译、版本和静态扫描证据见
[`report_artifacts/reproducibility.md`](report_artifacts/reproducibility.md)。
本次两种规定命令均退出 0，定理清单的行号已按通过编译的当前源文件重新生成。
文件还包含 202 条带 attribute 的辅助定理、10 条 `private theorem` 和 19 个可计算
`example`，它们与 472 条主要定理在同一次整文件编译中通过内核检查。
