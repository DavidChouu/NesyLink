# 环境形式化说明（技术附录 A）

## 1. 形式化对象与可信边界

环境与证明集中在
[`examples/NesyLinkEnvironment.lean`](examples/NesyLinkEnvironment.lean)。该文件以
Python 引擎和五个公开任务为语义基准，形式化 tile 级稳定状态、交互、完整
tick、搜索与闭环控制。它不证明 CNN 对任意 RGB 输入都正确，也不逐帧复刻浮点
AABB、动画或随机怪物 AI；这些边界通过 `ObservationRefinesWorld`、
`PixelBlockRefinement` 和公平性前提显式进入定理。

环境层与策略层的接口为：

```text
像素动作块满足运动学精化契约
            ↓
符号观测满足 ObservationRefinesWorld
            ↓
ControllerCommandSafe / CommandRefinesAction
            ↓
EngineTick / EngineExec
```

因此 Lean 证明的是“符号观测正确且像素动作块满足约定时，planner、安全层和
tile 执行满足规范”，而不是把神经感知正确性隐藏成无条件事实。

## 2. 基础类型、对象与世界状态

### 2.1 坐标、动作和方向

- `Position` 使用整数坐标，向北或向西越界会产生负数，再由 `inBounds` 排除。
- `Bounds` 保存房间宽高及正性证明。
- `Direction` 为 north、south、west、east。
- `Action` 为 wait、up、down、left、right、slotA、slotB，与测评接口的 0–6
  七种动作一致。
- `ObjectId` 和 `RoomId` 为对象与房间的稳定标识。

### 2.2 玩家与对象

| Lean 类型 | 关键运行时属性 |
|---|---|
| `PlayerState` | 位置、朝向、HP、最大 HP、inventory、举盾标志 |
| `Inventory` | keys、gold、items、A/B 装备槽 |
| `Chest` | ID、位置、loot、visible、opened、揭示条件 |
| `Monster` | ID、位置、种类、HP、接触伤害 |
| `Trap` | spike/abyss、伤害、重生点、激活与一次性标志 |
| `Button` | 位置与 pressed 状态 |
| `Switch` | 位置、目标房间与目标桥 |
| `Bridge` | 三组方向 tile 与当前公开桥朝向 |
| `FiniteBridgeCycle` | 任意非空有限模式表与当前模式索引 |
| `Npc` | 位置与对话文本 |
| `Exit` | 两个边界 tile、方向、目标房间/spawn、条件、opened、完成标志 |

`RoomState` 保存房间静态和动态对象；`WorldState` 保存有限 `roomIds`、
`RoomId → RoomState`、当前房间、玩家和完成状态。跨房间后，原房间的开箱、
清怪、机关和门状态仍保留。

`RuntimeWorld`/`RuntimeControl` 另行抽象装备动作窗口、盾牌有效窗口、怪物 stun、
abyss 控制锁和待重生状态。`runtime_windows_never_increase` 与
`abyss_lock_blocks_player_input` 验证其最基本的计时和控制锁约束。该 wrapper
没有被伪称为逐字段复刻 Python 运行时。

## 3. 出口、条件与目标

`Requirement` 支持：

- `.free`；
- `.keys count consume`；
- `.buttonPressed id`；
- `.ownsItem item`；
- `.allMonstersDefeated`；
- `.both left right`。

`Exit` 同时保存 `tileA`、`tileB`，所以两格宽门不会被简化成单点。锁门首次
通过时 `spendExitRequirement` 按配置消费钥匙并写回 `opened = true`；以后再次
通过不重复消费。完成型出口设置 `completed = true`。

目标语言包括 `HasKey`、`HasItem`、`ChestOpened`、
`AllMonstersDefeated`、`ButtonPressed`、`RoomReached`、`ExitReached` 和
`WorldCompleted`，并可用 `Goal.both` 组合。Task5 使用
`allWorldChestsOpened`：世界必须确实含有宝箱，且有限房间集中每个宝箱都已
可见并打开，零宝箱世界不会真空完成。

## 4. 物理可通行与 planner 安全

环境严格区分两个谓词：

```lean
canEnter  : RoomState → Position → Prop
safeTile : RoomState → Position → Prop
```

`canEnter`（亦记作 `physicalEnterable`）要求目标：

1. 在边界内；
2. 不是墙、NPC 或可见宝箱；
3. 不是未被活动桥覆盖的动态 gap。

怪物和普通陷阱不是物理墙，允许玩家与之接触并触发后续结算。可见宝箱即使
`opened = true` 仍是实体阻挡，这由
`opened_chest_is_still_a_static_blocker` 单独证明。

`safeTile`（亦记作 `plannerSafe`）在 `canEnter` 上进一步排除活动陷阱和怪物。
所以普通 BFS 不主动进入危险格，战斗、rush 或接触风险只能通过具有额外前置
条件的命令表达。

## 5. 交互与局部转移

### 5.1 几何和移动

交互使用 `interactionReach`，即曼哈顿距离不超过 1，覆盖同格和四邻接。
移动首先由动作确定方向；`moveBlocked` 在越界或遇到物理阻挡时保持位置、更新
朝向并清除临时举盾，因此可以忠实表达“先用短移动面向，再按 A”。

主要 `Step` 分支包括：

- `movePlain`：进入普通可通行格；
- `moveButton`：移动后踩下按钮；
- 存活/致死陷阱分支及 abyss 事件；
- `moveBlocked`：碰撞后停留；
- 宝箱、NPC、switch、剑攻击和无效果交互；
- 怪物移动、接触伤害、盾牌格挡；
- 出口通过、出口阻挡和全宝箱完成。

### 5.2 A 键优先级

`slotA` 按以下优先级建模：

```text
可见未开宝箱 → NPC → switch → 已装备剑 → 无效果
```

后续分支显式排除更高优先级交互，不能在相邻宝箱时任意选择攻击。宝箱开启后
根据 `Loot` 增加钥匙、金币、物品或治疗；治疗使用 `min maxHp`。

### 5.3 战斗与击杀结算

- `swordDamage = 1`；
- `monsterKillGold = 2`；
- 非致死攻击严格降低目标 HP；
- 致死攻击移除怪物并奖励 2 金币；
- 最后一只怪物死亡时，`resolveMonsterKill` 同步持久打开清怪条件门，并在全部
  房间中揭示满足 `ChestRevealCondition` 的隐藏宝箱。

怪物具体寻路和随机选择被抽象为受约束的非确定移动：每步仍须相邻、界内、
不进入物理阻挡且不与其他怪物重叠。盾牌只对当前有效窗口的接触结算生效。

### 5.4 按钮、switch、桥和 abyss

button 在移动后的 tile effect 中触发；switch 通过相邻 A 交互改变目标桥。
通用策略使用 `FiniteBridgeCycle`，只要求模式列表有限且非空，不硬编码三态。
公开 Task4 的 `BridgeOrientation` 为：

```text
westToNorth → westToEast → westToSouth → westToNorth
```

活动桥 tile 覆盖底层 abyss/gap；非活动方向保持不可进入。公共定理证明三次
旋转恢复原态，并证明从任一公开桥态到任一目标态至多两次旋转。

## 6. Tick、完整执行与不变量

`Step` 是便于局部证明的微步关系。`PlayerStep` 要求世界处于 `Running`，且本步
不是纯自主事件；`AutonomousStep` 表示怪物移动、接触、完成结算等自主微步。

```lean
EngineTick  = 一次 PlayerStep + 零次或多次 AutonomousStep
EngineExec  = EngineTick 的有限动作序列闭包
```

该类型保证“每个 tick 先有一次玩家输入，再有自主更新”，但没有用阶段索引强制
所有自主微步的唯一内部顺序；公开证书按预期顺序构造具体执行。这是相对于逐行
Python tick 的明确抽象边界。死亡或完成状态不能开始新的 `PlayerStep`。

### 6.1 基础状态不变量

`ValidState` 要求玩家位置界内且 `hp ≤ maxHp`。下列结论不依赖具体地图：

- `step_preserves_validState`；
- `engineTick_preserves_validState`；
- `engineExec_preserves_validState`；
- 合法移动不越界、不进墙、不进可见宝箱；
- 阻挡移动保持位置；
- 伤害不增加 HP，治疗不超过最大 HP。

### 6.2 完整世界不变量

`WorldInvariant` 组合：

- 有限非空且无重复的房间 ID；
- 当前房间有效；
- 所有对象和动态配置界内；
- 出口目标与 spawn 有效；
- 玩家不与物理阻挡重叠；
- 装备槽中的物品确实属于 inventory。

其主保持定理
`engineExec_preserves_worldInvariant_in_closed_system` 需要提供具体有限状态集
`worlds`、初态成员证据和 `TickInvariantClosed worlds`。这不是“任意坏地图经
执行自动变良构”，而是具体关卡必须提交的闭合可达状态证书。

## 7. BFS、路径证书与可执行性

通用搜索层定义有限邻接图路径、按深度 frontier 和可执行
`executableBfsWithin : ... → Option (List Position)`。主要证明为：

- 返回路径确实从起点出发、相邻并到达目标；
- 每个 tile 满足搜索给定的可通行谓词；
- 若给定深度内存在路径且 frontier 覆盖不变量成立，搜索返回某条路径；
- `DirectionPlanSafe`/`TilePath` 可翻译为 `Exec` 或 `EngineExec`；
- 公开路径还使用 exact 版本检查每一步终点和实际状态。

这里证明的是有界图搜索的健全性和条件完备性，没有把未证明的最短性写入报告。

## 8. 观测、命令与闭环

`VerifiedObservation` 保存玩家、可见对象、inventory、walkable 和 safe 谓词。
`ObservationRefinesWorld` 要求这些符号结果与真实世界一致。它是强而显式的感知
契约，不是 CNN 正确性定理。

`ControllerCommand` 区分普通导航、仅转向、开箱/NPC/switch 交互、攻击、举盾、
宝箱 rush、出口推进、memory fallback 和等待。`ControllerCommandSafe` 为每类
命令规定不同前置条件；`CommandRefinesAction` 保证最终动作等于声明动作。

`ClosedLoopStep` 同时携带：

1. 符号观测精化；
2. 命令安全；
3. 对应 `EngineTick`；
4. 控制器更新。

`closedLoopExec_projects_engineExec` 证明任意闭环策略轨迹都投影为真实环境
`EngineExec`。`closedLoop_complete_of_local_fair_progress` 使用自然数度量：若
每个非零状态均存在非空闭环执行使度量严格下降，且零度量推出目标，则最终存在
达成目标的闭环执行。
